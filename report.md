## 1. 需求分析

### 1.1 项目背景

现代社会中，年轻人在爱情与两性关系中面临大量情感困惑——沟通不畅、情绪波动、关系定位模糊等问题普遍存在。传统的心理咨询门槛高、费用贵，而简单的情绪关键词匹配又过于冰冷机械。本项目旨在构建一个兼具**专业情感分析能力**与**温暖人格化交互体验**的 NLP 应用，为用户提供一个随时可倾诉的"情感良师益友"。

### 1.2 功能点列表

| 编号 | 功能模块 | 功能点 | 优先级 |
|------|---------|--------|--------|
| F1 | 情感分析 | 对用户输入文本进行 18 种细腻情感标签识别 | P0 |
| F2 | 情感分析 | 输出情感类别（正面/负面/中性）及置信度 | P0 |
| F3 | 情感分析 | 提取文本中的情感关键词与次要情绪 | P1 |
| F4 | 良师益友 | 基于分析结果生成温暖、有洞察力的回应 | P0 |
| F5 | 良师益友 | 先共情后建议的对话风格 | P0 |
| F6 | 星座匹配 | 双星座恋爱匹配度计算（12×12 矩阵） | P1 |
| F7 | 星座匹配 | AI 深度解读匹配优势、挑战与建议 | P1 |
| F8 | 星座运势 | 单星座今日爱情运势生成 | P2 |
| F9 | 历史记录 | 对话历史持久化存储与查看 | P1 |
| F10 | 历史记录 | 历史记录清空功能 | P2 |
| F11 | 系统 | API 不可用时的自动降级运行 | P0 |
| F12 | UI | 三 Tab 切换、消息动画、响应式布局 | P1 |

### 1.3 用例分析

#### 用例 1：情感倾诉

```
参与者：用户
前置条件：用户打开应用
主流程：
  1. 用户在输入框输入情感相关文本
  2. 系统接收文本，调用情感分析服务

  6. 系统自动保存至历史记录
```


### 2.4 数据库设计

**数据库**：SQLite 3（轻量级、零配置、适合课程项目）

**表结构**：

```sql
CREATE TABLE IF NOT EXISTS history (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_text       TEXT    NOT NULL,                 -- 用户原始输入（限500字）
    sentiment_label TEXT,                             -- 主要情感标签（如"焦虑"）
    sentiment_category TEXT,                          -- 情感类别（positive/negative/neutral）
    sentiment_data  TEXT,                             -- 完整情感分析结果（JSON）
    ai_response     TEXT    NOT NULL,                 -- AI 回应文本
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP  -- 创建时间
);
```

**索引策略**：使用主键 id 自增索引 + created_at 降序查询。

**数据清理策略**：当记录数超过 `MAX_HISTORY`（默认 100 条）时，自动删除最旧的记录，保证存储空间可控。

### 3.1 关键代码实现

#### 3.1.1 DeepSeek API 封装层 (`deepseek_client.py`)

```python
class DeepSeekClient:
    """DeepSeek Chat API 客户端，封装了请求构建、错误处理、JSON 解析"""

    def chat(self, user_message, system_prompt="", temperature=0.7, max_tokens=2048):
        """通用对话接口 — 发送消息并返回文本响应"""

    def chat_json(self, user_message, system_prompt="", temperature=0.3):
        """结构化输出接口 — 要求模型返回 JSON，自动清理 markdown 代码块"""
```

**设计要点**：
- 全局单例模式 `get_client()` 避免重复初始化
- `chat_json()` 方法自动添加"只输出 JSON"指令，并处理模型可能包裹的 \`\`\` 代码块
- 多层 JSON 解析容错：直接解析 → 正则提取 → 异常抛出
- 超时设置 30 秒，防止请求阻塞

#### 3.1.2 情感分析服务 (`sentiment.py`)

**System Prompt 设计**（良师益友人格）：

```
你是一位名叫"白开水"的情感导师，你的定位是用户的良师益友。

## 你的性格
- 温暖、善解人意、有共情力
- 像一位经历过很多、想明白了的知心朋友
- 温柔但有洞察力，能看到用户自己没意识到的情感层面

## 你的风格
- 先说"我理解你"（共情），再说"你可以这样看"（洞察）
- 不要冷冰冰地贴标签或打分
- 像深夜和朋友聊天一样温暖自然
```

**情感标签体系**（18 种，面向爱情领域）：

| 类别 | 标签 |
|------|------|
| 正面 (positive) | 甜蜜、幸福、期待、感动、安心、满足、感恩 |
| 负面 (negative) | 失落、焦虑、委屈、吃醋、心碎、纠结、愤怒、不安、孤独 |
| 中性 (neutral) | 迷茫、回忆、思考、好奇、观望 |

**温度参数策略**：
- 情感分析：`temperature=0.3`（低温保证分析准确性）
- 回应生成：`temperature=0.8`（高温增加表达多样性和温度感）

**降级回应模板**：

```python
        "neutral":  "我能感觉到你此刻内心有些想法在酝酿..."

```

#### 3.1.3 星座匹配度矩阵 (`zodiac_data.py`)

使用 12×12 二维矩阵存储星座间匹配度（0–100 分），基于星座元素理论：
- 同元素星座通常匹配度高（如火象+火象）
- 互补元素有较好的化学反应（如水象+土象）
- 冲突元素需要更多磨合

匹配度等级划分：90+（天生一对）、80+（极佳匹配）、70+（良好默契）、60+（一般组合）、<60（需要磨合）。

## 5. 核心技术说明

### 5.1 Agent 记忆管理（历史记录存储方式）

本系统的"记忆管理"体现在**对话历史的持久化存储与检索**：

**存储方案**：SQLite 轻量关系数据库
- **选择原因**：零配置、嵌入式、无需独立服务进程，适合课程项目
- **表设计**：`history` 表包含用户原文、情感标签（label + category）、完整分析 JSON、AI 回应、时间戳
- **JSON 冗余存储**：`sentiment_data` 字段保存完整情感分析 JSON，为未来数据挖掘保留原始数据

**记忆管理策略**：
- **容量控制**：`MAX_HISTORY = 100`，超出自动删除最旧记录（FIFO）
- **查询能力**：按时间倒序排列，支持 limit 参数控制返回条数
- **隐私尊重**：提供清空功能，用户可随时删除全部记录

### 5.2 工具调用逻辑（API 调用架构）

本系统采用**分层 API 调用架构**：

```
sentiment.py / zodiac.py  →  deepseek_client.py  →  DeepSeek API
      (业务层)                  (基础设施层)           (外部服务)
```

**调用流程**：

1. **业务层**构造领域特定的 prompt（情感分析 prompt / 星座分析 prompt）
2. **基础设施层** `deepseek_client.py` 统一处理：
   - 请求构建（headers、payload）
   - 超时控制（30 秒）
   - 异常分类（Timeout / RequestException / JSONDecodeError）
   - JSON 响应清理（去除 markdown 代码块标记、正则容错提取）
3. **业务层**验证返回字段完整性，填充默认值

**容错设计**：每一层都有 try-except 降级逻辑，确保即使最外层 API 不可用，系统仍返回有意义的本地结果。

### 5.3 提示词设计思路（Prompt Engineering）

本系统的核心 NLP 能力通过 Prompt Engineering 实现，而非传统分类器模型：

#### 分层 Prompt 架构

| 层级 | Prompt | 温度 | 用途 |
|------|--------|------|------|
| System Prompt | 白开水人格定义 | — | 贯穿所有用户对话，定义 AI 角色 |
| 情感分析 Prompt | 情感标签体系 + JSON 指令 | 0.3 | 精确分析用户情感状态 |
| 回应生成 Prompt | 用户原文 + 情感结果 | 0.8 | 生成温暖对话回应 |
| 星座分析 Prompt | 星座元素 + 性格数据 | 0.7 | 专业星座解读 |

#### 设计原则

1. **角色锚定**：System Prompt 中明确定义姓名（白开水）、性格（温暖有洞察）、风格（先共情后建议）、边界（不评判、不极端建议）
2. **结构化输出**：需要机器可读的结果（情感分析、星座匹配），使用 `chat_json()` 强制 JSON 输出，辅以 markdown 清理和正则容错
3. **温度分层**：分析型任务使用低温（0.3）保证一致性，对话型任务使用高温（0.7-0.8）增加自然感
4. **领域知识注入**：星座分析 prompt 中注入星座元素、性格、恋爱风格等结构化知识，让模型基于事实而非幻觉生成内容

#### 关键 Prompt 示例

**情感分析 Prompt**（节选）：
> 请分析以下用户在爱情/情感方面的倾诉文本，识别其情感状态。
> 情感标签体系：正面情绪（甜蜜、幸福...）、负面情绪（失落、焦虑...）、中性情绪（迷茫、回忆...）
> 请以 JSON 格式返回：{"label": "...", "category": "...", "confidence": 0.0-1.0, "emotions": [...], "keywords": [...]}

**人格 System Prompt**（节选）：
> 你是"白开水"——温暖、善解人意的情感导师。先说"我理解你"，再说"你可以这样看"。不要冷冰冰贴标签，像深夜和朋友聊天一样自然。

### 5.4 前端技术选型理由

选择**原生 HTML/CSS/JS**而非 Vue/React 框架：

- **零构建步骤**：无需 webpack/vite 等工具链，直接刷新即可看到修改
- **课程项目适配**：减少依赖，降低环境复杂度
- **性能极致**：无框架运行时开销
- **单页应用模式**：通过 Tab 切换模拟 SPA，手写状态管理简单透明

### 5.5 降级运行策略

| 故障场景 | 检测方式 | 降级行为 |
|---------|---------|---------|
| 未配置 API Key | config.py 读取为默认值 | 跳过 API 调用，使用本地模板 |
| API 超时 | requests.Timeout 异常 | 捕获后返回降级数据 |
| API 返回格式异常 | JSONDecodeError | 正则提取兜底，失败则用默认值 |
| 必填字段缺失 | 字段存在性检查 | 补充默认值（"未知"/空列表/0.5） |

此策略确保系统在无网络、无 API Key、API 异常等情况下**始终可用、不崩溃**。
